    <article class="markdown-body entry-content container-lg" itemprop="text"><h1 dir="auto"><a id="user-content-不可恢复错误" class="anchor" aria-hidden="true" href="#不可恢复错误"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>不可恢复错误</h1>
<p dir="auto">Rust提供了一种被称为panic(故障)的机制，它会终止调用它的线程，而不会影响任何其他线程。宏<code>panic!</code>主要用于测试，以及不可恢复性(unrecoverable)错误。当 <code>panic!</code>宏被调用时，发生灾难性故障的线程开始展开函数调用堆栈，从调用它的位置开始，一直到线程入口点。</p>
<p dir="auto">为了找出导致灾难性故障的一系列调用，我们可以通过运行任何发生灾难性故障的程序，并在命令行设置环境变量 <code>RUST_BACKTRACE=1</code> 来查看线程中的回溯。</p>
<blockquote>
<p dir="auto">通常不可恢复性错误代表bug，或系统环境变量故障。</p>
</blockquote>

<p dir="auto">除直接调用外，以下场景也会引发panic!错误。</p>
<ul dir="auto">
<li>越界访问</li>
<li>被除以零</li>
<li>Option.unwrap/expect(可选值)</li>
<li>Result.unwarp/expect(返回值)</li>
</ul>
<blockquote>
<p dir="auto">unwrap和expect区别？</p>
<p dir="auto"><code>expect</code> 与 <code>unwrap</code> 的使用方式一样：返回<code>OK</code>或调用<code>panic!</code>宏</p>
<p dir="auto"><code>expect</code> 用来调用 <code>panic!</code> 的错误信息将会作为参数传递给 <code>expect</code> ，而不像<code>unwrap</code> 那样使用默认的 <code>panic!</code> 信息。</p>
</blockquote>
